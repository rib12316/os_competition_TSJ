## 0. 变更说明（v1 → v2）

v1 假设「通用 CUDA + vLLM」世界，按 `M1–M11` 组织模块，且模块之间存在叙事合并 / 前置依赖（M3+M10 合并、M7 依赖 M5、M5 跨三人）。真机栈落实后，三处根本修正：

| 修正 | 原因（实证） | 处置 |
|---|---|---|
| **硬件现实落地** | 实际栈是 **Ascend 910B2C / CUDA-free / vllm-ascend**，不是 CUDA/vLLM | 全文以 Ascend 为决策地基（见 [§硬件与栈现实](#硬件与栈现实决策地基)） |
| **作废两项** | ① SGLang 官方不支持 Ascend，radix 在 NPU 上跑不通（[#3609](https://github.com/sgl-project/sglang/issues/3609)/[#10841](https://github.com/sgl-project/sglang/issues/10841)）；② 910B 无 FP8 单元，`--kv-cache-dtype fp8` 会崩（[#8102](https://github.com/vllm-project/vllm-ascend/issues/8102)） | M5（SGLang 双引擎）整体作废；FP8 KV 改型为 **int8 KV** |
| **解耦** | v1 的 M 清单任务互相绑死，无法单人/单分支推进 | 重构为 **9 个独立功能 F1–F9**，落在 **7 条稳定缝**上，每个功能单主责、可独立交付（见 [§7 条稳定缝 + 9 个独立功能](#7-条稳定缝--9-个独立功能)） |

> **旧 M → 新 F 映射**（保连续性）：M1→F1（FP8 改 int8）· M2→F2 · M3→F5 · M4→F4 · M5→❌作废 · M6→F3 · M7→F7（降级为测量版）· M8→F9 · M9→低优（不入 F）· M10→F6 · M11→低优（不入 F）。新增 F8（多硬件对照，顶替 M5 的「适配不同框架/硬件」20 分）。

## Context

**赛题要求：** 基于 vLLM / llama.cpp 等开源推理框架扩展，面向 LLM agent 推理流程实现内存管理优化。在保证任务成功率基本不下降前提下，降低显存占用、提升推理效率，基于国产开源操作系统（openEuler/openKylin/OpenHarmony 至少一个）部署，用可复现 Benchmark 评估。

**设计原则：**
- 不从零造引擎，在 **vllm-ascend（V1 架构）之上**扩展
- **每个优化是一个独立功能**：自己的配置、自己的开关、走现成 harness 出自己的 before/after、可单独测、可单独交付
- 每个优化点都要有 before/after 数字证明
- 把 agent 推理的独特内存特征，映射到对应的优化机制

## 赛题核心解读（评分指挥棒）

| 评分项 | 权重 | 关键动作 |
|---|---|---|
| 功能完整性 | 30% | 跑得通的 agent 系统 + 至少一个内存优化点 + 多模型/多任务 |
| 应用效果 | 40% | **必须有 before/after 数字**：显存↓、延迟↓、成功率不掉 |
| 代码规范 | 20% | 模块化、单元测试、CI、Docker 一键复现 |
| 文档质量 | 10% | 技术报告 + 部署指南 + 测试报告 |

**应用效果 40% 细分（赛题原文，逐条命中）：**
- 保证成功率基本不下降前提下，显存↓ + 延迟优化（40 分）— 主战场
- 针对智能体长生命周期推理典型问题的有效方案（40 分）— 叙事差异化
- 适配不同模型规模或不同推理框架（20 分）— 通用性证明（本栈由 **F8 多硬件对照** 命中）

**Agent 推理 vs 普通推理的真正差异（决定优化点选择）：**
1. 系统提示 + 工具定义巨长且每轮重复 → prefix caching + Prompt 压缩（**F2**）
2. 多轮对话 KV 持续累积，单 session 寿命远超 single-shot → 生命周期管理（**F5/F6**）+ 分层存储（**F4**）
3. ReAct 推理可能产生分支（self-consistency / tree-search）→ 同前缀多分支共享（**F7**）
4. 工具返回值（HTML / 长 JSON / 图）一次性塞进 context → 工具数据按需加载（**F3**）
5. KV 显存绝对占用大 → KV 量化（**F1**，本栈为 int8）

## 硬件与栈现实（决策地基）

> 这是 v2 一切技术判断的前提。脱离这块现实做的方案都会在真机上崩。

| 项 | 实际值 | 对设计的影响 |
|---|---|---|
| 芯片 | **Ascend 910B2C**（Atlas A2，64GB HBM），`/dev/davinci15` | 910B 是 **BF16**，无 FP8 单元 → FP8 KV 作废 |
| 推理后端 | **vllm-ascend 0.22.1rc1**（editable，`third_party/vllm-ascend`） | Ascend 原生，非 CUDA vLLM |
| vLLM 内核 | vllm 0.22.1（editable，`third_party/vllm`） | V1 架构，默认 prefix cache，自带 ~12 个 V1 KV connector |
| CANN | 9.0.0 + 910b ops（9390 kernel） | 必须有 910b ops，否则算子执行崩 |
| torch | 2.10.0+cpu + torch-npu 2.10.0.post2 | CUDA-free，零 nvidia 包 |
| triton | triton-ascend 3.2.1（注册 `ascend` 后端） | |
| 本地模型 | 仅 `models/Qwen3-0.6B/` | dev/冒烟用它；真实 benchmark 仍需下 Qwen2.5-7B / MiniCPM3 |
| NPU 启停 | **默认停着，由用户控制** | 任何真机步骤前暂停等用户启动设备 |

**已实证可行的 Ascend 原生能力（这些是功能落地的靠山）：**
- vllm-ascend 自带 `simple_kv_offload/`（"NPU adaptation of vLLM's simple CPU KV-cache offloading"）+ `kv_offload/cpu_npu.py`（`torch.npu.Stream` 真 NPU↔CPU KV 传输）→ **F4/F5/F6 的机制现成**
- 已装 vllm 自带 V1 connector：`SimpleCPUOffloadConnector`、`lmcache_connector.py` + `lmcache_integration/`、`MultiConnector`、`P2pNccl` 等
- **LMCache-Ascend**（社区插件，2025-11）官方支持 NPU，`--enable-lmcache` 开启 → **F4 可行**
- KV cache 量化：Ascend 上 **int8**（非 FP8），在线动态量化无需校准 → **F1 待真机验证 0.22.1rc1 是否仍支持**（[#5630](https://github.com/vllm-project/vllm-ascend/issues/5630) 曾在 0.13rc1 移除）

## 7 条稳定缝 + 9 个独立功能

> **缝（seam）= 一个稳定接口位置。** 不同缝的功能天然独立；同缝内的功能靠接口隔离、可单独 toggle。这是 v2 解耦的核心。

### 独立性判定（让"独立"可操作）

一个功能算**独立**，当且仅当满足全部：① 有自己的 yaml preset，只开关自己；② 无前置模块依赖；③ 走现成 harness 出自己的 before/after，不改别的功能代码；④ 可单独测试；⑤ 可与别人组合，但其价值/证明不依赖别人。

### 缝清单

| 缝 | 定义 | 稳定接口 | 成员 |
|---|---|---|---|
| **缝A · 引擎 flag 预设** | 只改 `vllm serve` CLI flag | `EngineConfig.extra_args` | **F1**、prefix-cache 开/关 |
| **缝B · 引擎后端** | 换引擎/硬件 | `EngineConfig.backend` ∈ {vllm, vllm-ascend} | **F8** |
| **缝C · KV connector** | 换 KV 存储后端 | `--kv-transfer-config` + connector 名 | **F4** |
| **缝D · 上下文中间件** ⭐需新建 | 发引擎前变换 messages + 拦截工具结果 | 新接口 `Middleware.transform_messages()` / `.intercept_tool_result()` | **F2、F3** |
| **缝E · Session 生命周期策略** | 在 session 事件触发 offload/save | 策略模块，挂在 session manager | **F5、F6** |
| **缝F · 自带工作流的研究/demo** | 自包含 workload + 文档 | `Runner` Protocol（自带 ToT driver） | **F7** |
| **缝G · 部署** | 容器基底/OS | Dockerfile + compose service | **F9** |

> **唯一依赖关系（干净分层，非耦合）：** 缝E（F5/F6）的**机制**借缝C的 offload connector，但**策略**独立。F5/F6 各有自己的 yaml/触发逻辑/before/after，只把"KV 怎么搬到 CPU"委托给现成 connector——可对 stub offload 单独开发、单独测。

### 功能 → 缝 归属

```
F1 int8 KV 量化       → 缝A      F6 KV checkpoint/恢复  → 缝E
F2 Prompt 压缩        → 缝D ⭐   F7 分支 CoW（测量版）   → 缝F
F3 工具数据 lazy-load → 缝D ⭐   F8 多硬件对照            → 缝B
F4 LMCache 分层       → 缝C      F9 openEuler/Ascend 部署→ 缝G
F5 Session idle 淘汰  → 缝E
（⭐ = 需先补缝D 接口）
```

## 各功能详解

> 每个功能：得分锚点 / 缝 / 最简实现 / 开源借鉴 / Ascend 可行性 / 工作量。

### F1 — int8 KV 量化（原 M1，FP8 改型）
- **缝A · 得分锚点：** "显存占用有效降低"
- **最简实现：** `--kv-cache-dtype int8`（一行 flag + 一份 yaml），在线动态量化无需校准
- **开源借鉴：** [vLLM quantized KV docs](https://docs.vllm.ai/en/stable/features/quantization/quantized_kvcache/)；本地 `OSTQuant/`、`SINQ/`、`FlatQuant/` 做差异化
- **Ascend 可行性：** ⚠️ **待真机验证（高风险）**。int8 是 Ascend 上 KV 量化的唯一支持格式，但 #5630 称曾在 0.13rc1 移除，0.22.1rc1 是否恢复未知。**建议第一件事就上 NPU 验**（最便宜探针：起 vllm-ascend 加该 flag 看是否报错）。若不行，降级为 max-num-seqs/gpu-mem-util 调参顶替"显存↓"叙事
- **工作量：** 验证 0.5 天 + benchmark 1 天

### F2 — Prompt / 上下文压缩（原 M2）
- **缝D · 得分锚点：** "Prompt 与上下文压缩"（赛题原文）
- **最简实现：** 包一层 `transform_messages`，对 system prompt / 冷历史用压缩器压 2–4×，红线上验成功率（下降 ≤ 2pp）
- **开源借鉴：** [microsoft/LLMLingua](https://github.com/microsoft/LLMLingua)（+ LLMLingua-2，task-agnostic、低延迟）；[AutoGen 集成示例](https://microsoft.github.io/autogen/0.2/docs/topics/handling_long_contexts/compressing_text_w_llmligua/)
- **Ascend 可行性：** ✅ 纯 agent 层，与硬件无关（需先补缝D 接口）
- **工作量：** 2–3 天 · **难度中**（token 收益 vs 成功率 trade-off）

### F3 — 工具调用数据 lazy-load（原 M6）
- **缝D · 得分锚点：** "工具调用数据优化"（赛题原文，对应最直接）
- **最简实现：** 拦截工具返回值（长 HTML/JSON/图）→ 存 SQLite/内存 → context 只放 `<doc id=.. summary=..>`，加 `fetch(id)` 工具让模型按需取
- **开源借鉴：** mem0 / LangChain retriever 的 reference-ID 模式；τ-bench 的订单/航班 JSON 是天然真实数据
- **Ascend 可行性：** ✅ 纯 agent 层（需先补缝D 接口）
- **工作量：** 3–4 天 · **难度中**（要设计引用语法 + few-shot 引导模型主动 fetch）

### F4 — LMCache 分层（GPU→CPU→Disk）（原 M4）
- **缝C · 得分锚点：** "分层内存与异构存储优化"（赛题原文）
- **最简实现：** `--enable-lmcache` + 一份 `LMCACHE_CONFIG_FILE` yaml（NPU↔CPU↔Disk 三级）
- **开源借鉴：** [LMCache/LMCache-Ascend](https://github.com/LMCache/LMCache-Ascend)（社区插件）；[部署指南](https://docs.vllm.ai/projects/ascend/en/latest/user_guide/feature_guide/lmcache_ascend_deployment.html)；本仓已带 `lmcache_integration/`
- **Ascend 可行性：** ✅ 官方插件 2025-11 发布，NPU 支持
- **工作量：** 配置 + 调试 1–2 天

### F5 — Session idle eviction（原 M3）
- **缝E · 得分锚点：** "长生命周期推理任务的动态资源回收与重分配"（赛题原文）
- **最简实现：** session idle N 秒 → 把 GPU KV 块 offload 到 CPU（机制给定），只写策略层
- **开源借鉴：** **vllm-ascend 原生 `simple_kv_offload/`** + 上游 `SimpleCPUOffloadConnector`
- **Ascend 可行性：** ✅ 机制现成（`kv_offload/cpu_npu.py` 真 NPU↔CPU 传输），只补 idle 策略
- **工作量：** 2–3 天

### F6 — KV checkpoint / 恢复（原 M10）
- **缝E · 得分锚点：** "长生命周期推理"的差异化叙事
- **最简实现：** 进程退出前 `save_kv_layer` 落盘；重启冷启动按 session_id 反向加载
- **开源借鉴：** 同 F5 机制（`SimpleCPUOffloadConnector.save_kv_layer` 已存在）+ SharedStorageConnector 思路
- **Ascend 可行性：** ✅ 与 F5 共机制，但**功能独立**（运行时淘汰 vs 跨重启持久化，v2 彻底拆开，不再合并叙事）
- **工作量：** 2–4 天

### F7 — 分支 KV 共享（测量版）（原 M7，降级）
- **缝F · 得分锚点：** "分支推理内存共享"（赛题原文）
- **最简实现：** 构造分支 workload（self-consistency / tree-of-thought），**测量 vLLM APC 的隐式 block 级 CoW 共享**（block 引用计数复用），而非造新 fork API
- **开源借鉴：** vLLM [Automatic Prefix Caching](https://docs.vllm.ai/en/stable/design/prefix_caching/)；workload 借 [Tree of Thoughts](https://github.com/princeton-nlp/tree-of-thought-llm)
- **Ascend 可行性：** ✅ 测量版可行；❌ "显式 fork 引擎改动"版不可行（vLLM 无公开 fork API，tree-attention 只用于 spec decode，[#3960](https://github.com/vllm-project/vllm/issues/3960)/[#18327](https://github.com/vllm-project/vllm/issues/18327)）
- **工作量：** 1 周+ · **难度高**（自带 ToT driver + 构造对照）

### F8 — 多硬件对照（替代作废的 SGLang 双引擎）
- **缝B · 得分锚点：** "适配不同模型规模或不同推理框架"（20 分）
- **最简实现：** 同一 agent loop + 同一 yaml，`backend: vllm`(CPU) vs `vllm-ascend`(NPU) 各跑一遍对比
- **开源借鉴：** 本仓已装的两套引擎本身
- **Ascend 可行性：** ✅ 两套都已装好（install-status 全绿）；几乎零成本
- **工作量：** 1 天

### F9 — 国产 OS / 硬件部署（原 M8）
- **缝G · 得分锚点：** 赛题"开源 OS"硬性要求（至少一个国内开源 OS）
- **最简实现：** openEuler = 用 CANN 的 openeuler24.03 镜像作基底 + 一份 deploy 脚本；Ascend = 本栈已是 vllm-ascend
- **开源借鉴：** [openeuler cann 镜像](https://hub.docker.com/r/openeuler/vllm-cpu)；[openEuler + vLLM-Ascend 部署 Qwen3](https://www.openeuler.org/zh/blog/2025-05-06-Qwen3/2025-05-06-Qwen3)
- **Ascend 可行性：** ✅ Ascend 部分已就绪；openEuler 容器化是"换基底镜像 + 写脚本"
- **工作量：** 1 天

## 验证方案

1. **冒烟：** `python -m agent_mem.cli "..."` 工具调用正常；τ-bench retail 样例多轮通过
2. **三档递进 + 逐功能对照：** 每个 F 功能 = baseline → 开启该 F，各 3 次取中位数，自动产 `logs/_summaries/<date>_<fX>_comparison.md`；门槛：显存峰值↓ ≥ 30%、延迟↓ ≥ 20%、成功率差 ≤ 2pp（不达标标红）
3. **多模型：** 同 F 在 Qwen2.5-7B 和 MiniCPM3-4B 各跑一遍，证优化方法通用
4. **多硬件（F8）：** 同 agent loop 在 vllm(CPU) vs vllm-ascend(NPU) 各跑一遍
5. **国产化（F9）：** openEuler 容器中跑通完整 benchmark
6. **代码质量：** `pytest --cov` ≥ 60%；`ruff check` 无错；CI 全绿

## 风险与注意事项

1. **F1 是最大未知：** int8 KV 在 910B/0.22.1rc1 待真机验证，可能不支持 → 尽早上 NPU 探针；备选降级方案
2. **NPU 启停由用户控制：** 默认停着，任何真机步骤（F1 验证、F4/F5/F6 联调、F9 容器内 benchmark）前**暂停等用户启动设备**
3. **benchmark 公平性：** 同硬件/同模型/同 prompt/同 seed，before/after 各 3 次取中位数；V1 默认开 prefix cache，baseline 对照显式关
4. **成功率红线：** 任何优化不得使 τ-bench 成功率下降 > 2pp（赛题硬指标）；压缩类（F2/F3）尤其要验
5. **多模型评分项：** Qwen + MiniCPM 各跑全套，证优化方法通用
6. **代码规范：** pyproject / ruff / pytest / CI / Docker 一键启动 — 20 分基础
7. **文档：** 技术报告含架构图 + 特征↔机制↔功能映射表 + 优化前后对比表 + 复现步骤；建议 mkdocs 站点
8. **国产化叙事：** openEuler 是最低成本达标路径，**不要漏做**（F9）
